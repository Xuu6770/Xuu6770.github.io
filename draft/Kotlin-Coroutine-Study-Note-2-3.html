线程的 stop()

Java 线程中的stop()也是老生常谈的内容了，它的作用是强制停止一个正在运行的线程。强制停止会带来包括但可能不限于以下问题：

  1. 线程突然终止会导致资源无法正常释放；
    1. 数据库连接没有正常关闭；
    2. 文件流没有及时关闭；
    3. 数据写入操作可能只完成一部分
  2. 突然终止会导致对象处于不一致状态。例如一个转账操作，可能转出账户已扣款，但转入账户还未收到款项就被终止了。
  3. 会立即释放该线程所持有的所有锁，可能导致受保护的数据处于不一致状态。
  4. ……

stop()由于本身的工作机制导致使用时会产生各种不可预期的后果，所以它在 1998 年时就已被废弃（deprecated），只不过考虑到兼容性，目前 JDK 里仍然保留着。

interrupt()

interrupt()是 Java 中推荐使用的线程中断机制，它属于「交互式」的中断，即它不会立刻杀掉线程，而是会给线程发送一个中断信号，让线程进行收尾工作后，自行结束。相比stop()来说，interrupt()更可靠，但这种可靠，并不是说其内部实现有多么优秀，同样的，stop()被废弃也不是因为它有多烂，这两个方法的处境是由它们自身的机制造成的，一个是强制中断，一个是交互式中断（非强制）。

另外需要知道，interrupt()是在线程外部被调用的，因为在线程内部调用这个方法并没有什么意义，如果想要在线程内部结束线程，那直接 return 就行了。

interrupt()虽说是交互式「中断」，但是到底中不中断，中断前要干什么，也不是它说了算，它只是起到一个通知的作用而已。进一步说，它的通知方式有两种，一种是修改中断标记位，第二种则是抛出异常，这里先来看第一种。

第一种情况是当interrupt()被调用时，调用它的线程的中断标记位就会被置为 true ，此时如果线程内有判断标记位的代码以及相应的线程收尾工作代码（这些代码都是开发人员自行编写的），则代表该线程可以配合中断标记位对线程进行中断处理。

第一种情况还算好理解，这其中需要注意的是「应该在线程内的什么位置来对中断标记位进行判断」——一般来说是在耗时操作之前进行判断，因为如果线程被通知该中断了，那么在耗时操作前及时中断线程就可以避免不必要的等待和资源浪费。

这里还要补充一个上面的示例代码中没有提到的方法：interrupted()，它和isInterrupted一样是用来判断标记位的，不同在于前者在判断完以后还会顺带把标记位置为 false 。也就是说假设你在线程外调用了interrupt()，然后在线程内连续调用多次interrupted()，那么只有第一次调用时会获得 true 结果，后续都会得到 false 。

第二种情况其实也不复杂，就是在线程执行阻塞操作（例如wait()、sleep()、join()）时调用了interrupt()的话，线程就会立即抛出InterruptedException。以sleep()来说，这就是一个简单的延时等待操作，但是在调用这个方法时，我们会被要求使用 try 语句来将sleep()包住。这就是因为在线程等待的过程中，一旦外部调用了线程的interrupt()方法，线程就会通过抛出异常的方式从睡眠中“醒”过来，此时如果要像在第一种情况中那样进行一些收尾工作并结束线程的话，就需要使用 catch 来捕获异常，同时在代码块中编写相应的收尾工作的代码。

如果对这两种情况进行分析，我们会发现其实第一种情况也是属于非强制性打断，因为interrupt()只是改了一个变量（标记位），然后需要我们自己在线程中去对变量进行判断。但是第二种情况则相反，属于强制性打断。这其实也好理解，因为第一种情况下存在太多未知因素，外部调用interrupt()时我们并不知道线程目前的工作进行到哪一步了，故而不能强制打断，否则也有可能会损坏程序的状态。但是在第二种情况下，线程是处于一个「等待」、「睡眠」的状态，此时线程并不会干其它的事情，只是在等而已，所以当外部调用interrupt()时，线程就会被直接唤醒（打断）。

对于第二种情况，还有两点补充一下，一是当外部调用interrupt()时，哪怕线程还没有开始睡眠（或等待），或者是即将开始睡眠（或等待），这都不重要，只要它一睡，异常就会被抛出。第二点是，异常被抛出后，线程的中断标记位也会被重置为 false 。
