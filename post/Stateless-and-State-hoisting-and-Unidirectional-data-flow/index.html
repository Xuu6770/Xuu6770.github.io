无状态、状态提升和单向数据流

记录一下学习 Compose 中遇到的几个概念。

无状态

Compose 是无状态的（Stateless），此处的“状态”意指控件属性，例如传统的TextView中的文字内容，可以通过getText()获取，通过setText()设置。

相对的，Compose 中的文本控件则没有这样的用法，即没有手段能够获取到文本控件的文本，这就是所谓的“无状态”。但是准确来说，无状态指的是无“内部”状态而已，一个文本控件本身没有文本内容的属性，但是其终究需要我们提供一个内容让它显示：Text(text)，此处的text是一个变量，文本控件将显示这个变量中的内容，对于控件来说，这个变量就是它的状态，可以简单理解为内部的状态被提取到了外部。

状态提升

“无状态”也不是绝对的，例如一个 Composable 函数中定义了一个变量，那么该函数就是有状态的。可就算它有状态，我们也无法做到像传统TextView一样去获取它的状态，因为这个状态是一个局部变量。

这时候如果想要灵活地去控制该变量的值，可以将这个变量作为该函数的一个参数，让调用方在调用函数的时候根据需求传入参数。

这种写法被称为“State Hoisting”，即“状态提升”。

单向数据流

以TextField()为例。

TextField()不属于 foundation / ui 层而是属于 material 层，因为它不仅是一个文字输入框，而是和Button()一样，它是一个有着 Material Design 风格的输入框。如果想要使用基础样式的输入框，可以使用 foundation 层的BasicTextField()。

TextField()默认接收两个参数，其中第一个参数value: String表示显示在编辑框中的文本，TextField()将原本应该是自身属性的字符串作为参数供调用者传入，这是一种状态提升的体现。

第二个参数是编辑框内部文本改变的监听器，是一个函数类型的参数，可以写成 Lambda 表达式的形式。这个参数需要我们自己去编写，如果不写，用户在键入内容时，编辑框不会有任何变化。之所以系统不自动帮我们更新编辑框内容（用户输入什么你就显示什么就行了），而是提供这么一个回调，是因为 UI 组件不应该对外部造成不可预期的影响。这样的做法体现了 Compose “单一信息源”的原则。再一来，将变量（value）往内部去传递，将事件（onValueChange）往外部传递，这样数据下、事件上的形式形成了一种闭环，这样的闭环是 Compose 所倡导的设计模式——Unidirectional Data Flow（单向数据流）。同样的理念和行为还可以在Checkbox等 Composable 上看见。整体来说，单向数据流的做法帮助我们实现了单一信息源。
