线程同步和线程安全

volatile

通过一个例子来简单理解volatile关键字。

在 1 处设置了一个标记位，在 2 处开启一个线程，主要内容就是一个 while 循环，循环条件就是标记位。接着在 3 处给主线程睡了 1 秒钟，在 4 处把标记位改为false。程序运行的理论效果应该是在 1 秒钟后结束，因为在 1 秒钟后修改了 while 循环的条件使其停止循环，子线程结束从而整个程序正常结束。

但实际运行效果并不是这样。实际运行起来程序并不会自己结束，这是为什么呢。

出于性能考虑，线程修改一个变量是先把它复制到自己的内存区域，对其进行处理，然后再返回到原来的地方。以上面这个例子来说，子线程拿到了running，并把它作为标记为去执行循环操作，然后主线程又拿了一份running，并把它改为false然后返回。此时running的值确实是false，但是问题就出在子线程所持有的running仍然还是一开始拿到它的时候的值，也就是true，就是说变量的更新并没有同步到子线程中，导致循环没有停止，程序无法结束。

解决这个问题就需要引入volatile关键字，用它来修饰会被多个线程读写的变量，尽管这么做降低了效率，但是保证了同步性和正确性。将变量声明改为volatile boolean running = true;，重新运行程序就可以看到程序在 1 秒后自动结束。

原子性

首先在 1 处，创建了一个整型变量，并且为了防止同步性问题，使用了volatile关键字对其进行修饰。在 2 处开启了一个线程，循环一万次让 x 加 1 ，在 3 处也开启了一个线程并且逻辑和 2 处的一样。如此一来，理论上两个线程中肯定会有一个最终打印的 x 是大于或等于 10000 的，然后另一个线程打印的就一定是 20000 。

但实际运行起来在大多数情况下没有一个线程能输出 20000 ，偶尔可能有一次一个线程 10000 另一个 20000 ，这是为什么呢。

因为x++;这行代码并不是一个原子操作，所谓「原子操作」可以理解为代码的执行过程不会被打断，或者说在执行时不会切换线程，以x++;为例，这看似只有一行的代码实际上是由两步操作组成，一步是int temp = x+1;，然后才是x = temp;。这样一来，当多个线程都在执行x++;的时候，有可能线程 1 刚执行完第一步，虚拟机就切换到线程 2 去执行第一步了，从结果来看两个线程拿到的 x 的值是相同的，但是实际上我们想要的结果是线程 2 拿到的应该是经过线程 1 处理过（+1）的 x ，而不是没处理完的 x 。

解决方案就是引入synchronized关键字，用它来修饰plusOne()，使这一整个方法具有原子性，从而就算出现了线程切换，也会先等这个方法执行完了再切换。另外，使用synchronized关键字以后，x 将不再需要volatile修饰。

除了引入synchronized以外，还可以通过AtomicInteger来解决，AtomicInteger从名字上来看就是一个带有原子性的整型，但除了原子性，它还带有同步性。使用AtomicInteger将不再需要synchronized修饰方法，也不需要volatile修饰变量：private AtomicInteger x = new AtomicInteger(0);。相应的，也不再能使用x++;和++x;这种写法，而是替换成x.getAndIncrement();和x.incrementAndGet();。从输出结果来看，线程 1 和线程 2 最终输出的 x 的值都会是 20000 ，这应该就是同步性的体现。

乐观锁和悲观锁

乐观锁和悲观锁属于数据库方面的问题，简单来说就是先从数据库中拿数据，进行一些处理，然后再写回数据库中，但是这时候有可能会遇到数据库中的数据已经被别人修改了的情况，如果这时候往回写就属于覆写，所以一般是写回去之前先做个判断，如果数据和原来的不一样，那就要拿出最新的数据进行处理然后再写回去，如此重复。当然这也只是有可能而已，并不是一定，所以这种「可能会被别人改掉数据」的情况就叫「乐观锁」（乐观并发控制），虽然它并没有锁（往回写的时候可能会加锁）。

与此相反的，在读之前就上锁，读写完后才释放锁的行为就叫「悲观锁」。

最简单的单例类一般这么写就够了，但是在多线程环境下这么写是不够的。考虑到有可能同时存在两条线程，它们同时执行到达 1 处，随后它们又一起进入到 if 语句的内部也就是 2 处去执行，线程 1 执行一次，拿到了一个实例化的对象，线程 2 又去执行一次，也拿到了一个实例化的对象，但是线程 2 在实例化对象的时候会把 1 实例化好的结果给替换掉，两个线程拿到的 instance 并不是同一个。而且在这之后，程序都只会拿线程 2 实例化好的对象，因为在这之后，别的线程来访问 if 语句时，结果已经为 false 了，而线程 1 就一直拿着一个过时的对象。

异步消息处理

Android 中的异步消息处理主要由 4 个部分组成：

  • Message：Message类实现了Parcelable接口，类中定义了几个属性，包括整型的what、arg1、arg2和Object类型的obj，这些属性可以携带少量的信息，用于在不同线程之间传递数据。
  • Handler：主要用于发送和处理消息。发送消息一般使用sendMessage()、post()等，而发出的消息经过一系列地辗转处理后，最终会传递到handleMessage()中。
  • MessageQueue：主要用于存放所有通过 Handler 发送的消息。这部分消息会一直存在于消息队列中，等待被处理。每个线程中只会有一个 MessageQueue 对象。
  • Looper：调用 Looper 的loop()后就会进入一个无限循环当中，然后每当发现 MessageQueue 中存在一条消息时，就会将它取出，并传递到 Handler 的handleMessage()中。每个线程中只会有一个 Looper 对象。

线程间通信

interrupt()

一个线程启动另一个线程和一个线程终结另一个线程是多线程中常见的操作，其中如何终止一个线程作为基础知识时常被提起。启动一个线程使用start()，就算根据方法名来猜，都知道停止一个线程应该用stop()——当然也确实有这么个方法，只不过已经被废弃了。stop()确实可以用来终止线程，但是因为它终止线程的手段过于强硬，以至于会造成不可预期的后果，所以被废弃。比如说，stop()调用的时候线程可能正在进行一些变量赋值的操作，因为线程被掐掉了，导致赋值操作被中断，整个系统处于一个中间状态或者说不稳定的状态。再比如说，有一些资源在使用完或者不再使用时需要手动释放，但是因为线程被打断了，导致资源没能被释放，这也不合适。

综上所述，终止线程需要一个更好的方案（一个手段不那么强硬的方案），故而引入interrupt()。「interrupt」中文大意为「打断」，其最终目的也是终止线程，只不过它会更温和一些，采用类似于通知的行为：将线程的中断标记位置为true，以此告知线程为终止做准备。interrupt()不是立刻终止线程，也不是强制终止线程，线程自己本身需要适配interrupt()，例如线程在循环执行任务时，需要对中断标记位进行判断（最好在耗时操作前进行判断），当标记位为true，那么线程就需要做一些准备工作（比如上面提到的把资源进行释放），然后返回。也就是说如果线程不适配，那么interrupt()调用也是白调用。

wait() / notify()

某个线程在 synchronized 范围内因为暂时未满足某些条件而主动调用wait()，此时这个线程将会让出这个 synchronized 范围的访问权（让出锁）并将自己放入一个等待区。而notify()则会在将来某一个时刻当条件满足以后由别的线程调用，调用后会在等待区随机出队一个线程，出队后的线程并不会马上获得锁，如果此时存在其它线程也想要锁的话，出队的线程会和这些线程公平竞争。因为notify()是随机出队一个线程，存在不确定性，为了稳定起见，一般会使用notifyAll()让所有的线程出队。

wait()和notify()/notifyAll()一般是同时出现，如果只有wait()，那进入等待区的线程还真就再也出不来了。另外wait()和notify()/notifyAll()属于 Object 类，它们都是在方法内被调用，而方法往往都会被synchronized修饰。如果是在 synchronized 代码块内被调用，那么在调用的时候还要加上调用者（否则就会抛出IllegalMonitorStateException），例如：

  • onPreExecute()：在后台任务开始执行前调用，用于进行一些初始化操作。比如显示一个进度条对话框等。
  • doInBackground()：这个方法中的所有代码都会在子线程中运行（意味着应该在这里去处理耗时任务，并且不能操作 UI）。任务一旦完成，就可以通过return语句将任务的执行结果返回，如果Result参数指定的是Unit，就可以不返回任务执行结果。
  • onProgressUpdate()：这个方法会在后台任务调用了publishProgress()后接着被调用，该方法中携带的参数就是在后台任务中传递过来的。在这个方法中可以对 UI 进行操作，利用参数中的数值就可以对界面元素进行相应的更新。
  • onPostExecute()：当后台任务执行完毕并通过return语句返回时，这个方法就很快会被调用。返回的数据会作为参数传递到此方法中，可以利用返回的数据进行一些 UI 操作，比如说提醒任务执行的结果，以及关闭进度条对话框等。

内存泄漏？

简单来说，内存泄漏指的是一块内存已经无用了但是却无法被释放，或者说无法被垃圾回收器回收。又或者说，内存泄漏也可以认为是一个被存储在内存中的对象无法被代码访问到。

网传继承了 AsyncTask 的内部类需要用static修饰，否则这个类就会持有外部 Activity 的引用，导致 Activity 在销毁时仍然无法被回收，从而发生内存泄漏。但是从更加通用的角度来看，并不是所有的内部类都会造成内存泄漏，所以内存泄漏的原因应该不是因为持有引用，而是正在执行的线程不会被回收，从而线程指向的对象就也不会被回收了。因为垃圾回收器只会去回收那些 GC Root 没有直接或间接引用的对象（引用不可达），所谓 GC Root 指的是：

  • 由系统类加载器加载的类，也包含对静态变量的引用。
  • 存储在本地堆栈中的方法的变量和参数。
  • 处于活动中的线程。
  • JNI 引用，为 JNI 调用创建的 Native 代码 Java 对象，包含本地变量、JNI 方法的参数和全局 JNI 引用。（JNI References: Native code Java objects created for JNI calls; contains local variables, parameters to JNI methods, and global JNI references.）

话说回来，AsyncTask 确实会造成内存泄漏，但是只要 AsyncTask 的执行时间不要太长，就可以不用做防⽌内存泄露的处理。况且除了 AsyncTask 以外的方案，例如 Thread ，同样有泄漏的风险。

该用哪个？

  • AsyncTask 已经在 API level 30 也就是 Android R（11）被废弃，证明它已经没有使用的必要了。

  • 通过 Thread 来开启线程也已经是过时的方案，因为它不易于管理。

  • Executor：能用就用。用完以后需要关闭，例如在 Activity 的onDestroy()里调用 Executor 的shutdown()。

  • HandlerThread：在实际应用中和 Executor 相⽐并没什么优势，而且它还是单线程的，除了在要指定主线程执行代码的场景以外还不如用 Executor 。用完以后需要关闭，例如在 Activity 的onDestroy()里调用 HandlerThread 的quit()。
