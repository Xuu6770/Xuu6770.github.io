更多的 Kotlin 学习笔记：协程

如果只是了解基础语法，那只能算是入门了 Kotlin 。

历史

Kotlin 的协程是 2017 年初在 1.1 版本加入进来的，那时协程还只是实验性的（Experimental）。等到 2018 年底，Kotlin 更新到 1.3 版本的时候，协程才成为 Kotlin 的正式特性。接着到 2019 年 Kotlin 协程才推出 Flow 相关的 API。事实上早在 1967 年的 Simula 语言当中，就已经出现了协程。不过在之后的几十年里，协程并没有被推广开，后续的 C、C++、Java 之类的语言，更多的是使用线程来进行异步和并发。直到 2012 年左右，C# 重新拾起了协程这个特性，实现了 async、await、yield。之后，JavaScript、Python、Kotlin 等语言才继续跟进实现了对应的协程。

概念

  • 可以简单理解为「相互协作的程序」，例如在编写代码时，可以通过yield()挂起，让出执行权，然后当下一个需求到来时再继续执行。
  • 可以理解为更加轻量的线程，成千上万个协程可以同时运行在一个线程当中，但是成千上万个线程就可能会出现内存溢出。
  • 可以理解为是运行在线程当中的轻量的 Task ，这些 Task 支持挂起和恢复，当 Task 挂起时，就不会阻挡到后续的 Task 。
  • 不会与特定的线程绑定，可以在不同的线程之间灵活切换。
  • 是非阻塞式的，而线程是阻塞式的，比如协程的delay()和线程的Thread.sleep()。

在 Kotlin 中，可以用suspend关键字来修饰函数，被修饰的函数将成为“挂起函数”。挂起函数可以以编写同步代码的方式完成异步任务，是 Kotlin 协程最大的优势。

  • 使用挂起函数后，表面上看起来是同步的代码，实际上涉及到了线程切换。例如上面的val user = getUserInfo()这一行代码就切换了两个线程。其中等号左边的代码运行在主线程，而等号右边的代码运行在 IO 线程。
  • 每一次从主线程到 IO 线程，都是一次协程挂起。每一次从 IO 线程到主线程，都是一次协程恢复。挂起和恢复是挂起函数特有的能力。
  • 挂起只是将程序执行流程转移到了其他线程，主线程不会被阻塞。如果代码运行在 Android 系统，那么 App 仍然可以响应用户的操作，主线程并不繁忙。

  • 把挂起函数反编译至 Java 后会变成一个普通函数，只不过这个函数会额外接收一个Continuation类型的参数。
  • Continuation 封装了挂起函数的执行状态，当一个函数被挂起时，其相关信息会被保存到 Continuation 中，然后将该 Continuation 传递给异步操作，当异步操作完成后，将会使用该 Continuation 对象来恢复挂起函数的执行。
  • Continuation是一个接口，包含了：
  • Continuation 和回调类似，但不完全一样。
    • 控制流（指程序执行的顺序和流程）方向：回调的控制流方向是由异步操作来控制的。在使用回调函数的场景中，我们通常将回调函数传递给异步 API ，异步 API 在异步操作完成后会调用回调函数，从而改变控制流的方向。而 Continuation 是由我们自己来控制控制流方向的，它通过协程来管理挂起函数的状态。协程可以随时暂停执行，并在异步操作完成后恢复执行。
    • 上下文信息：回调函数通常只包含异步操作完成后的结果或错误信息。而 Continuation 不仅包含了结果或错误信息，还包含了协程的上下文信息，例如协程的堆栈和局部变量等。
    • 处理方式:在使用回调函数的场景中，我们通常需要编写一些额外的代码来处理异步操作的结果或错误信息。而在使用协程的场景中，我们可以像编写同步代码一样编写异步代码，避免了大量的回调函数嵌套。
  • 从挂起函数到 Continuation 的过程叫做“CPS 转换”（Continuation-Passing-Style Transformation）。协程之所以是非阻塞，是因为它支持“挂起和恢复”，而挂起和恢复的能力，主要是源自于“挂起函数”，而挂起函数是由 CPS 实现的。
  • 综上所述，挂起函数只能在协程作用域中调用，或者是被另一个挂起函数调用。
    • 以runBlocking()这个函数来说，尽管可以在它的 Lambda 表达式中调用挂起函数，但实际上它的 Lambda 表达式的本体就是一个被suspend修饰的函数类型的参数，所以说到底，作用域中调用挂起函数其实还是挂起函数调用挂起函数。
    • 关于为什么只能在挂起函数中调用挂起函数，是因为挂起函数中包含一个Continuation类型的参数，而Continuation当中又包含了只有在协程作用域里才能传递的协程上下文。

协程作业（Job）

在编程领域中，句柄（Handle）通常指一个标识符或引用，它可以用来引用或访问操作系统、应用程序中的对象或资源。协程的句柄通常是Job类型的对象，它提供了一些函数来管理协程的状态和执行，通过 Job 暴露的 API 还可以让不同的协程之间互相配合，从而实现更加复杂的功能。

CoroutineContext

CoroutineContext 可以理解为协程的上下文，是一个用于定义协程行为和配置的关键概念，在很多地方都有用到：

  • launch()中的第一个参数，接收的就是 CoroutineContext 对象，之前一直没传递是因为它有默认值。
  • withContext()的第一个参数也是 CoroutineContext 对象，而当我们在使用withContext()时，会经常传入Dispatchers.Main、Dispatchers.IO等值，由此可知，这些值也都是协程上下文。同时，Dispatchers也被称为“调度器”，是一个单例类，用于确定协程应在哪个线程或线程池上运行。
  • runBlocking()的第一个参数也是 CoroutineContext 对象，所以在使用时也可以向其传入Dispatchers.IO等值。

Dispatchers中除了IO以外，还有其它的可选值，这些可选值当中，除了Main以外的都是CoroutineDispatcher类型，而Main则是MainCoroutineDispatcher类型，不过MainCoroutineDispatcher也是继承自CoroutineDispatcher。这些可选值为：

  • Dispatchers.Default：这是一个用于 CPU 密集型任务的默认调度器。它在一个共享的线程池中运行协程，该线程池的线程数量根据设备的 CPU 核心数量进行调整。如果需要在协程中执行密集型计算，可以使用此调度器。
  • Dispatchers.Main：这是一个在 Android 和其他 UI 框架（如 JavaFX 和 Swing）中使用的主线程调度器。它用于执行与 UI 相关的操作，如更新 UI 组件。当需要在协程中执行 UI 操作时，可以使用此调度器，也就是说，该调度器在普通的 JVM 工程当中是无法直接使用的。
  • Dispatchers.Unconfined：这是一个特殊的调度器，通常仅用于测试和调试目的，不建议在生产环境中使用，其原因在于该协程可能运行在任意线程之上。
  • Dispatchers.IO：这是一个用于 I/O 密集型任务的调度器，它使用了一个专门的线程池来处理 I/O 操作，如文件读写、网络请求和数据库操作等。这个线程池的线程数量根据系统可用的处理器数量进行调整，以避免由于过多的线程导致的性能问题。值得注意的是，当 Dispatchers.Default 线程池中有富余线程的时候，它是可以被 IO 线程池复用的。

CoroutineScope

CoroutineScope 是启动协程的入口，提供了协程的上下文信息，定义了协程的基本操作，包括启动协程、取消协程、处理异常等。CoroutineScope 还可以与 Activity 或 Fragment 的生命周期相关联（例如在 Activity 或 Fragment 的onStop()中取消协程），这样一来，在 Activity 或 Fragment 的生命周期结束时，CoroutineScope 的生命周期也会结束，而所有在该 CoroutineScope 中启动的协程都会被自动取消。

要点：

  • Channel()实际上并不是一个构造函数，而是一个带有泛型参数和另外三个参数的顶层函数。
  • capacity: Int：代表了 Channel 的容量。
  • onBufferOverflow: BufferOverflow：指定 Channel 在数据溢出时的处理方式：
    • SUSPEND：默认值，缓冲区满了以后如果还要继续发送，发送方就会被挂起。
    • DROP_OLDEST：丢弃最旧的那条数据，然后发送新的数据。
    • DROP_LATEST：丢弃当前正准备发送的那条最新数据，而管道中的内容将维持不变。
  • onUndeliveredElement: ((E) -> Unit)?：当某些数据发送了但是没有被成功接收时会调用这个回调。

可以看到，当 Channel 的容量是无限大时，发送方可以一直往 Channel 中塞入数据，等数据都塞完以后，接收方才开始接收。如果此处的 capacity 是默认值的话，那么程序执行后，发送和接收的操作将会交替执行。

而当把 capacity 设置为Channel.CONFLATED时，程序运行又会有别的结果：

main @coroutine#2 → 发送消息：1
main @coroutine#2 → 发送消息：2
main @coroutine#2 → 发送消息：3
main @coroutine#3 → 接收消息：3

对于该选项，如果有新的元素发送到该 Channel ，它将替换掉已经缓存的元素，否则新发送的元素将被直接缓存。如果不进行这个设置，而是将 capacity 设置为1，然后把 onBufferOverflow 设置为BufferOverflow.DROP_OLDEST，也可以得到一样的效果。

而当把 onBufferOverflow 设置为BufferOverflow.DROP_LATEST并且 Channel 的容量已经满了情况下，新发送的数据会被直接丢弃。

还有就是Channel.cancel()和Channel.close()的区别：前者用于取消当前 Channel 的所有挂起操作，包括发送和接收，但是在这种情况下，Channel 仍然可以重新发送和接收数据；后者则会把 Channel 整个关闭，不再允许发送数据，但是已经发送到 Channel 中的数据仍然可以被接收。

Channel 中还有两个属性isClosedForReceive和isClosedForSend，它们分别用来帮助接收方和发送方判断当前 Channel 是否已经关闭。

Flow

Flow 提供了一种声明式的、响应式的数据流处理方式，用简洁、易读的代码处理异步操作，而无需回调或其他复杂的同步方法。

「Flow」意为「流」，这里可以理解为「数据流」。在 Flow 中，存在发送数据的「上游」和接收数据的「下游」，以及在数据传输中进行各种处理的「中转站」。

终止操作符

终止操作符或者又叫末端操作符，该类操作符一般用于将 Flow 转换为一个最终的数据结果或触发一个副作用。首先来看launchIn，该操作符用于将flowOn作用域以外的数据流放到一个指定的协程作用域中执行：

输出结果：

emit: 1 in DefaultDispatcher-worker-1 @coroutine#3
emit: 2 in DefaultDispatcher-worker-1 @coroutine#3
emit: 3 in DefaultDispatcher-worker-1 @coroutine#3
map: 1 in main @coroutine#2
map: 2 in main @coroutine#2
map: 3 in main @coroutine#2

可以看到，除了 flow 代码块以外的部分，都是运行在launchIn()所指定的作用域当中的，包括 collect 中的代码——可是这里并没有调用collect操作符……这实际上是因为调用collect收集数据这件事已经由launchIn()替我们完成了，从它的源码中就可以得知。

此外还有很多其它的终止操作符：

  • first()：返回 Flow 中的第一个元素，如果 Flow 为空，则抛出异常。
  • single()：返回接收到的第一个元素，如果 Flow 发送了多个元素，则抛出异常。
  • fold()：对 Flow 发送的整型数据进行累加，返回最终结果：

输出结果：

emit: 1 in main @coroutine#1
map: 1 in main @coroutine#1
acc = 0, value = 2, sum = 2
emit: 2 in main @coroutine#1
map: 2 in main @coroutine#1
acc = 2, value = 4, sum = 6
emit: 3 in main @coroutine#1
map: 3 in main @coroutine#1
acc = 6, value = 6, sum = 12

fold()接收两个参数，第一个是一个整型，代表初始值；第二个是一个挂起的函数类型的参数，这个函数又有两个参数，acc代表累加值，value代表 collect 收集的每个值。其中acc默认等于初始值。

  • reduce()：和fold类似，只不过不需要手动指定初始值，而是使用 Flow 发送的第一个元素作为初始值。
  • toList()：将收集到的元素放到一个 List 中并返回。

上面这些操作符都有一些共同特性，首先是它们都会作为整个流处理中的最后一个操作符出现，并且在它们之后不能再添加别的操作符。其次就是它们都是挂起函数。
