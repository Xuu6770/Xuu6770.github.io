一点点 WPF DataBinding 的笔记

虽然 WPF Tutorial 上的数据绑定讲得很浅显，不过暂且还是记录一下。

Binding 的语法

最基础的形式：<TextBlock Text="{Binding}" />，表明本控件的Text属性的值将参与绑定，所有的工作都在花括号当中进行。

Path指定了需要和谁的什么属性进行绑定：<TextBlock Text="{Binding Path=Text}" />。基于Path是默认的属性，故而可以将其省略，但是需要注意，要把Path的值放在最前面：<TextBlock Text="{Binding Text, ElementName=tb}" />。

ElementName属性用于直接和某个元素进行关联，在上方的代码例子当中，本文块关联了名为“tb”的文本框。

DataContext

DataContext在FrameworkElement类中定义，除非手动指定别的绑定源，例如ElementName，否则DataContext将在绑定当中作为默认源。并且由于DataContext是通过控件层级向下继承的，所以可以把Window的DataContext设置成它自己，随后在内部的所有子控件中都可以去使用。

如此一来，便不需要在每个文本框内部去单独设置绑定源，只需要设置Path属性的值就能绑定成功了。第一个文本框内部显示的内容即为窗体的标题，并且当内容改变时，窗体的标题也会随之改变，第二、三个文本框同理。但这并不意味着所有的控件都只能使用从上面继承下来的DataContext，这种继承关系可以在任何时候的任何层级中被打断。此外这里还有一个现象就是，文本框中修改后的值并不会立刻应用到界面上，而是要将鼠标的焦点移出当前文本框以后界面才会更新。

UpdateSourceTrigger

在本文所基于的参考资料中，作者将文本框称为“destination control”，而窗体的标题、宽度、高度这三个根据文本框的内容而改变的属性被作者称为“source”，原因是这些“source”是会根据文本框中的内容而改变，也就是任意文本框中的内容实际上就是控件对应属性的目标，所以文本框会被成为“destination control”。

在 DataContext 结束时提到的界面不会立刻更新的问题，实际上与一个叫做UpdateSourceTrigger的属性有关。

3 个文本框使用了UpdateSourceTrigger属性的 3 个不同的值：

  • Explicit：按需更新界面，故而增加了一个按钮并且设置了相应的逻辑用于应用更新。
  • LostFocus：在文本框（destination control）失去焦点以后更新。
  • PropertyChanged：实时更新。

实际上UpdateSourceTrigger的默认值为Default，也就是说，属性的UpdateSourceTrigger的值应该是它该有的值，或者说，应该是由系统来决定（系统的决定会更合理），Default不是一个具体的选项，而是“视情况而定”，它可能是PropertyChanged，也可能是LostFocus。比如对于Text属性来说，它默认的UpdateSourceTrigger的值为LostFocus，这其实也很合理，因为文本框的内容更新实际上是非常频繁的，如果其绑定了一些复杂、耗时的操作，那么频繁执行这些操作必然会带来性能开销甚至可能会让程序崩溃。

响应数据改动

假设有一个ListBox，然后将一个List赋予它的ItemsSource，默认情况下当List里的内容发生变化时，ListBox的显示并不会随之改变。而响应数据改动的需求是，无论这个集合当中是增加还是删除项，甚至是项的属性发生了变化，ListBox都要响应到位。

解决方案分两步来实现，第一步是将List改成ObservableCollection，如此一来，集合当中的元素增删都会同步到ListBox当中。

第二步则是为相应的类实现INotifyPropertyChanged接口，这一步是为了集合中的项的属性修改能够应用到ListBox中。在当前例子中，需要为User类来实现这个接口，如果不执行这一步，假设修改了users中任意一项的Name，ListBox上对应的显示项是不会更新的。

使用 IValueConverter 对值进行转换

假设有一个多选框和一个文本框，现在的需求是多选框选中时，文本框显示否定的内容，反之显示肯定的内容。问题在于多选框的选中状态接受的是个布尔值，而文本框的文本接受的是个字符串类型，两个不同的类型如何交互，这就需要用到转换器。

布局部分需要注意声明<Window.Resources>，然后还要给需要绑定的控件的属性添加绑定对象ElementName=自定义, Path=自定义和转换器Converter={StaticResource 自定义}。

代码部分，转换器类需要实现IValueConverter或者IMultiValueConverter接口，然后实现Convert()以及ConvertBack()方法，前者是文本框内容变动到界面显示过程的中间人（destination control -> source），后者则是处理与之相反的流程，即点击多选框，文本框的内容发生变化（source -> destination control）。此处附上 WPF 内置的转换器的合集：converters - Built-in WPF IValueConverters - Stack Overflow。

Debugging 数据绑定

程序只有在运行时才会对数据绑定进行评估，并且就算出错了，程序也不会抛出异常。尽管如此，一般情况下还是可以从 Visual Studio 的“输出”窗口得到一些相关信息，得知是在哪里出现了什么错误，比如输错了绑定对象的名称。
